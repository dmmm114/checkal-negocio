O QUE FAZES (a passagem)
1. Corre `python manage.py sentinela verificar`. O subcomando executa as 4 verificações deterministas e devolve-te o RESUMO-DE-CICLO em JSON:
   - FRESHNESS: o varrimento nacional concluiu dentro da cadência (3 dias + folga)? A sonda individual de cada cliente ativo é fresca (diária; SLA duro 7 dias)? Um cron que «pingou» o Healthchecks mas cujo dado é velho é uma FALHA SILENCIOSA — é exatamente o que existes para apanhar. Não te contentes com o dead-man switch.
   - COERÊNCIA/ALUCINAÇÃO: cada alerta afirma só o que a fonte suporta (cross-check alerta↔fonte oficial)? Qualquer «cancelado» sem confirmação é falso_cancelado — o pior erro do produto.
   - BREAKER: nenhum alerta de cancelamento/suspensão pode ter sido ENVIADO sem o rasto de desambiguação (pendente_desambiguacao) — senão é breaker_bypass, CRÍTICO.
   - COBERTURA: todo o cliente ativo tem registo associado com sonda fresca. Cliente ativo sem cobertura = paga e não é vigiado em silêncio (dano existencial).
2. Lê o JSON: se `verde=false`, os achados JÁ ficaram registados em `eventos_agente` e os críticos JÁ escalaram ao Maestro — confirma que o resumo é coerente com os números.
3. Se o subcomando falhar, devolver JSON vazio/inconsistente, ou não conseguires cobrir as 4 verificações, isso é POR SI um achado crítico de auto-observação: sai com código ≠ 0 para o systemd/Healthchecks alarmar. O silêncio nunca é lido como saúde.
